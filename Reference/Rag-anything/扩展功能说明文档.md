
### 1.2 Whisper 模型下载

Whisper 模型首次调用时自动下载到 `~/.cache/whisper/`，无需手动操作。

模型大小与精度对照:

| 模型 | 参数量 | 显存需求 | 英文识别率 | 多语言识别率 |
|------|--------|----------|-----------|-------------|
| `tiny` | 39M | ~1GB | 低 | 低 |
| `base` | 74M | ~1GB | 中低 | 中低 |
| `small` | 244M | ~2GB | 中 | 中 |
| `medium` | 769M | ~5GB | 高 | 中高 |
| `large-v3` | 1.55B | ~10GB | 很高 | 很高 |

安装必要依赖:

```bash
# 安装 whisper + PyTorch (CPU 版)
uv sync --extra audio

# 如需 GPU 加速，手动安装 CUDA 版 PyTorch:
# https://pytorch.org/get-started/locally/
```

切换模型: 在 `.env` 中设置

```env
ASR_MODEL=base     # 轻量模型, 快速识别
ASR_MODEL=large-v3 # 最高精度 (默认)
```

## 三、接入新的 ASR 后端实现

只需创建一个继承 `AsrBackend` 的类，用 `@register_asr_backend()` 装饰即可。

### 3.1 最小实现示例

```python
# my_asr_backend.py
"""Azure Speech 服务后端"""

from raganything.asr import AsrBackend, AsrResult, register_asr_backend


@register_asr_backend()
class AzureSpeechBackend(AsrBackend):
    """Azure Speech Services ASR 后端."""

    binding = "azure"

    def __init__(self, config):
        super().__init__(config)
        self.model_name = config.asr_model or "zh-CN"
        self._client = None

    @property
    def identity(self) -> str:
        return f"azure:{self.model_name}"

    @classmethod
    def validate_config(cls, config) -> list[ValidationIssue]:
        issues = []
        if not config.asr_api_key:
            issues.append(ValidationIssue(
                level="error",
                field="asr_api_key",
                message="Azure subscription key required",
            ))
        if not config.asr_base_url:
            issues.append(ValidationIssue(
                level="error",
                field="asr_base_url",
                message="Azure region endpoint required",
            ))
        return issues

    def _warmup(self):
        import azure.cognitiveservices.speech as speechsdk
        speech_key = self.config.asr_api_key
        service_region = self.config.asr_base_url
        speech_config = speechsdk.SpeechConfig(
            subscription=speech_key,
            region=service_region,
        )
        speech_config.speech_recognition_language = self.model_name
        self._client = speechsdk.SpeechRecognizer(
            speech_config=speech_config,
        )

    def _do_close(self):
        self._client = None

    def transcribe(self, audio_path: str) -> AsrResult:
        self._acquire_op()
        try:
            self._ensure_warmup()
            self._safe_hook(self.before_transcribe, audio_path)

            import azure.cognitiveservices.speech as speechsdk
            audio_config = speechsdk.AudioConfig(filename=audio_path)
            recognizer = speechsdk.SpeechRecognizer(
                speech_config=self._client.speech_config,
                audio_config=audio_config,
            )

            # 同步等待识别完成
            result = recognizer.recognize_once()

            if result.reason == speechsdk.ResultReason.RecognizedSpeech:
                transcript = result.text.strip()
            else:
                transcript = ""

            segments = [{
                "start": 0.0,
                "end": float(result.duration) / 10_000_000 if result.duration else 0,
                "text": transcript,
                "confidence": None,
            }]

            ret: AsrResult = {
                "transcript": transcript,
                "segments": segments,
                "language": self.model_name,
                "duration": segments[-1]["end"] if segments else 0,
                "metadata": {"provider": {"model": self.model_name}},
            }
            self._safe_hook(self.after_transcribe, audio_path, ret, 0)
            return ret
        except Exception as e:
            self._safe_hook(self.on_error, audio_path, e)
            raise
        finally:
            self._release_op()
```

### 3.3 接入 Checklist

**必须实现的方法:**

| 方法 | 说明 |
|------|------|
| `binding` | 唯一标识字符串，对应 `ASR_BINDING` 环境变量 |
| `identity` (property) | 缓存键，格式 `{binding}:{model}:{endpoint_hash}` |
| `transcribe(audio_path)` | 同步转录方法，返回 `AsrResult` (base class 负责线程安全) |

**可选覆盖的方法:**

| 方法 | 说明 |
|------|------|
| `_warmup()` | 惰性初始化 (加载模型/建立连接)，首次 transcibe 时自动调用 |
| `_do_close()` | 释放资源 (卸载模型/关闭连接池)，**必须幂等** |
| `atranscribe()` | 原生异步转录 (如 HTTP API 直接 await) |
| `validate_config()` | 启动时验证 `asr_api_key`/`asr_base_url` 等配置完整性 |
| `capabilities` (property) | 声明 `AsrCapability.ASYNC` 等能力 |
| `before_transcribe()` / `after_transcribe()` / `on_error()` | 观测 hook |

**必须遵循的契约:**

- `transcribe()` 内必须调用 `self._acquire_op()` / `self._release_op()` (finally 块)
- `transcribe()` 内先调用 `self._ensure_warmup()` 再执行实际转录
- 使用 `self._safe_hook()` 封装 hook 调用，防止 hook 异常中断主流程
- 输出必须归一化为 `AsrResult` 格式
- API key 和 endpoint 使用 `self.config.asr_api_key` / `self.config.asr_base_url` (不 fallback 到 LLM 配置)
- 额外参数通过 `self.config.asr_backend_options` 字典传递

**注册规则:**

- 用 `@register_asr_backend()` 装饰器注册到全局注册表
- `binding` 不可与已有后端重名 (默认抛错，可传 `override=True` 覆盖)
- 确保模块在 RAGAnything 初始化前被 import (添加到入口文件的 import 中)

## 五、常见问题

### Q: "whisper not found" / 模型下载失败

手动下载模型到本地:

```bash
# 查看 whisper 缓存目录
python -c "import whisper; print(whisper._MODELS)"

# 手动下载 (以 large-v3 为例)
# 从 https://huggingface.co/openai/whisper-large-v3 下载模型文件
# 放入 ~/.cache/whisper/ 目录
```

### Q: FFmpeg 未找到

确认 FFmpeg 在 PATH 中:

```bash
ffmpeg -version
```

Windows 用户重启终端使 PATH 生效，或手动添加 `C:\ffmpeg\bin` 到系统环境变量。

### Q: 切换后端后缓存失效

缓存键 (`asr_identity`) 每个后端自动不同。切换 `ASR_BINDING` 或 `ASR_MODEL` 后旧缓存自动失效，无需手动清除。

### Q: 高并发场景卡住

参见 `AsrBackend.close()` 文档: 永久阻塞语义。若有 GPU hang 风险，在可控超时内调用 `close()`:

```python
import threading

def safe_close(rag, timeout=30):
    t = threading.Thread(target=rag.close)
    t.start()
    t.join(timeout)
    if t.is_alive():
        print(f"警告: close() 在 {timeout}s 内未完成")
```

## 六、视频解析扩展

以下给出两个方案：**方案二（原生 Video 类型）** 为推荐方案，完全仿照音频的接入模式；**方案一（Parser 拆分）** 为轻量备选。

---

### 6.1 参考音频模式的接入架构

音频的接入贯穿 **7 个层**，视频方案二会完全照搬每一层：

| 层 | 音频对应文件 | 视频将新增/修改 |
|---|---|---|
| 格式声明 | `parser.py` — `AUDIO_FORMATS` | `parser.py` — `VIDEO_FORMATS` |
| Parser 网关 | `parser.py` — `parse_audio()` (仅记录路径) | `parser.py` — `parse_video()` (仅记录路径) |
| Config | `config.py` — `enable_audio_processing` 等 | `config.py` — `enable_video_processing` 等 |
| 后端抽象 | `raganything/asr/` — `AsrBackend` + 注册表 | 复用现有 ASR 后端；关键帧提取不需要后端抽象 |
| ModalProcessor | `modalprocessors.py` — `AsrModalProcessor` | **新文件** `raganything/video_processor.py` — `VideoModalProcessor` |
| 管道集成 | `processor.py` — 扩展名路由到 `parse_audio()` | `processor.py` — 扩展名路由到 `parse_video()` |
| 内容路由 | `utils.py` — `_PROCESSOR_KEY_MAP["audio"] = "audio"` | `utils.py` — `_PROCESSOR_KEY_MAP["video"] = "video"` |
| Chunk 模板 | `prompt.py` — `audio_chunk` | `prompt.py` — `video_chunk` |

**核心对称性**：Parser 层保持轻量，只做路径记录；全部复杂处理在 Processor 层完成。

---

### 6.2 方案二（推荐）：原生 Video 类型

完全仿照 `AsrModalProcessor` 的模式。Parser 层仅记录视频路径，新增 `VideoModalProcessor` 处理全部逻辑。

#### 数据流

```
视频.mp4  →  parse_video()
              │
              └── {"type": "video", "video_path": "...", "file_name": "..."}
                    │
                    ↓
              VideoModalProcessor.generate_description_only()
                    │
                    ├── 1) FFmpeg 提取音频 → 调用 ASR backend 转录
                    │      ↓
                    │      AsrResult (已有 AsrBackend 体系)
                    │
                    ├── 2) FFmpeg/OpenCV 提取关键帧 → VLM 描述
                    │      ↓
                    │      frame descriptions (复用 VLM 或直接调用 llm_model_func)
                    │
                    ├── 3) 时间对齐合并
                    │      ↓
                    │      每个时间段 = 转录片段 + 对应关键帧描述
                    │
                    ├── 4) MapReduce → 视频级摘要 + 实体提取
                    │      ↓
                    │      生成 entity_info
                    │
                    └── 5) _segment_with_overlap() → 多 chunk
                           ↓
                           每个 chunk 含：时间戳、transcript、帧描述、说话人标签
```

#### 与 AsrModalProcessor 的对称对照

| AsrModalProcessor | VideoModalProcessor |
|---|---|
| `_load_or_transcribe()` → ASR 转录 | `_load_or_analyze()` → ASR 转录 + 关键帧提取 |
| `_segment_with_overlap()` → 按 token 分段 | `_segment_with_overlap()` → 按 token + 关键帧分段 |
| `_recursive_mapreduce()` → 摘要 | `_recursive_mapreduce()` → 摘要（含帧描述） |
| `AUDIO_ENTITY_TYPE_ENUM` | `VIDEO_ENTITY_TYPE_ENUM` |
| 单一 transcript → N chunk | 帧描述 + transcript → N chunk（时间对齐） |

#### Chunk 设计

每个 chunk 包含一个时间段的所有信息：

```python
PROMPTS["video_chunk"] = """## Video Segment [{start_time:.0f}s–{end_time:.0f}s]

- **File**: {file_name}
- **Time range**: {start_time:.1f}s – {end_time:.1f}s
- **Segment**: {segment_index}/{total_segments}
- **Key frames**: {frame_descriptions}

**Transcript**:
{transcript}
"""
```

### 6.3 方案一（轻量备选）：Parser 拆分

当不需要视频级统一分析（只需分别索引音频内容和关键帧内容）时采用。零新增 Processor，改动量极小。

#### 数据流

```
视频.mp4  →  parse_video()
              │
              ├── FFmpeg 提取音频 → {"type":"audio", "audio_path":"...", "file_name":"..."}
              ├── FFmpeg 提取关键帧 → {"type":"image", "img_path":"frame_001.jpg", "page_idx": 0}
              │                        {"type":"image", "img_path":"frame_002.jpg", "page_idx": 1}
              └── 混合 content_list
                    │
                    ↓
              已有 multimodal 管道自动分流
                    ├── audio → AsrModalProcessor (复用 ASR 后端)
                    └── image → ImageModalProcessor (复用 VLM)
```

#### 改动范围

| 文件 | 改动 |
|---|---|
| `parser.py` | 添加 `VIDEO_FORMATS` + `parse_video()` 含 FFmpeg 提取逻辑 |
| `config.py` | `enable_video_processing`、`video_keyframe_interval` |
| `processor.py` | 添加扩展名路由到 `parse_video()` |
| 执行依赖 | FFmpeg（已安装），无需新 Python 包 |

#### 局限性

- 音频和关键帧之间的**时间关联丢失**（无法知道某关键帧对应哪段语音）
- 无法生成视频级摘要

### 6.4 接入步骤说明（方案二）

#### Step 1: 格式声明 + Parser 网关

**在 `parser.py` 的 `Parser` 类中添加：**

```python
VIDEO_FORMATS = {
    ".mp4", ".avi", ".mov", ".mkv", ".flv", ".wmv",
    ".webm", ".m4v", ".mpg", ".mpeg", ".3gp",
}
```

**添加 `parse_video()` 基类方法（与 `parse_audio()` 对称）：**

```python
def parse_video(
    self,
    video_path: Union[str, Path],
    output_dir: Optional[str] = None,
    **kwargs,
) -> List[Dict[str, Any]]:
    """Parse video file — minimal: just records the path. All processing
    is done by VideoModalProcessor.

    Supported formats: {VIDEO_FORMATS}

    Returns:
        Single-item content list with type "video"
    """
    video_path = Path(video_path)
    if not video_path.exists():
        raise FileNotFoundError(f"Video file does not exist: {video_path}")

    ext = video_path.suffix.lower()
    if ext not in self.VIDEO_FORMATS:
        raise ValueError(
            f"Unsupported video format: {ext}. "
            f"Supported formats: {', '.join(sorted(self.VIDEO_FORMATS))}"
        )

    return [{
        "type": "video",
        "video_path": str(video_path.resolve()),
        "file_name": video_path.name,
    }]
```

#### Step 4: VideoModalProcessor

**新建 `raganything/video_processor.py`，完全仿照 `AsrModalProcessor` 的结构设计：**

```
VideoModalProcessor
├── __init__(lightrag, modal_caption_func, asr_backend, config, tokenizer)
├── process_multimodal_content()        # 入口，兼容已有管道
├── generate_description_only()          # 主流程
│   ├── _load_or_analyze()              # 缓存管理
│   │   ├── _extract_audio()            # FFmpeg 提取音频（新）
│   │   ├── _transcribe_audio()         # 复用 ASR backend
│   │   └── _extract_keyframes()        # FFmpeg/OpenCV 提取帧（新）
│   ├── _describe_keyframes()           # VLM 描述帧（新）
│   ├── _align_audio_with_frames()      # 时间对齐（新）
│   ├── _segment_with_overlap()         # 仿照音频分段
│   └── _recursive_mapreduce()          # 仿照音频 MapReduce
└── close()                             # 释放资源
```

**核心主流程方法签名参考：**

```python
async def generate_description_only(self, modal_content, content_type,
                                    item_info=None, entity_name=None):
    """主入口：提取音频 → ASR → 提取关键帧 → VLM 描述 → 时间对齐 → 摘要"""
    video_path = modal_content.get("video_path")
    if not video_path:
        raise ValueError(f"No video_path in modal_content: {modal_content}")

    # 1. 检查缓存
    cached = self._load_cached_analysis(video_path)
    if cached:
        return cached["summary"], cached["entity_info"]

    # 2. 提取音频轨道
    audio_path = await self._extract_audio(video_path)

    # 3. ASR 转录（复用 AsrBackend）
    asr_result = await self._transcribe_audio(audio_path)

    # 4. 提取关键帧
    keyframes = await self._extract_keyframes(video_path)

    # 5. VLM 描述关键帧（复用 VLM 调用）
    frame_descriptions = await self._describe_keyframes(keyframes)

    # 6. 时间对齐：将帧描述分配到对应时间段的 transcript 中
    aligned_segments = self._align_audio_with_frames(
        asr_result["segments"], frame_descriptions
    )

    # 7. 分段 + MapReduce（仿照 AsrModalProcessor）
    segments = self._segment_with_overlap(aligned_segments)
    global_summary, entity_info = await self._recursive_mapreduce(
        segments, entity_name, item_info, video_path, asr_result
    )

    # 8. 缓存结果
    await self._persist_analysis(video_path, global_summary, entity_info)
    return global_summary, entity_info
```

**缓存策略：** 完全仿照音频缓存模式（`_load_cached_analysis` / `_persist_analysis`），使用文件大小 + 修改时间 + SHA-256 + 配置哈希做三层验证。缓存键额外包含 `video_keyframe_interval` 和 `video_max_frames`。

**分段策略：** 参考 `AsrModalProcessor._segment_with_overlap()`，增加帧描述合并逻辑。

### 6.5 视频解析的技术建议

#### 音频轨道提取

使用 FFmpeg（已在音频部分安装）：

```python
import subprocess

def extract_audio(video_path: str, output_path: str):
    """Extract audio track from video file."""
    cmd = [
        "ffmpeg", "-i", video_path,
        "-vn",                    # 忽略视频流
        "-acodec", "pcm_s16le",   # 16-bit PCM WAV
        "-ar", "16000",           # 16kHz 采样率（Whisper 推荐）
        "-ac", "1",               # 单声道
        "-y",                     # 覆盖输出文件
        output_path,
    ]
    subprocess.run(cmd, check=True, capture_output=True)
```

#### 关键帧提取

**方式 A：FFmpeg 场景检测（推荐，零额外依赖）**

```bash
# 每 N 秒提取一帧
ffmpeg -i video.mp4 -vf "fps=1/10" -frame_quality 75 frame_%03d.jpg

# 或使用场景变化检测（更智能，避免冗余帧）
ffmpeg -i video.mp4 -vf "select='gt(scene,0.4)',showinfo" -vsync vfr frame_%03d.jpg
```

**方式 B：OpenCV（需要 `opencv-python`）**

```python
import cv2

def extract_keyframes(video_path: str, interval: int = 10) -> List[str]:
    """Extract keyframes at regular intervals."""
    cap = cv2.VideoCapture(video_path)
    fps = cap.get(cv2.CAP_PROP_FPS)
    frame_interval = int(fps * interval)
    keyframes = []
    frame_idx = 0

    while True:
        ret, frame = cap.read()
        if not ret:
            break
        if frame_idx % frame_interval == 0:
            path = f"frame_{frame_idx}.jpg"
            cv2.imwrite(path, frame)
            keyframes.append(path)
        frame_idx += 1

    cap.release()
    return keyframes
```

### 6.7 常见问题

### Q: 视频处理依赖哪些外部工具？

方案二依赖 FFmpeg（音频提取 + 关键帧提取），这部分已在音频扩展中安装。关键帧的 VLM 描述直接使用系统已有的 `llm_model_func`，无需额外依赖。

### Q: 长视频（>1 小时）如何处理？

参考音频的 MapReduce 策略：
1. 按 `video_keyframe_interval` 分段提取关键帧
2. ASR 分段按 token 限制做 overlap chunking
3. 每个 chunk 独立做描述
4. Recursive reduce 逐层合并

`video_max_frames` 控制关键帧上限，超过上限后均匀采样。

### Q: 如何确保缓存有效性？

视频分析结果缓存策略与音频相同（文件大小 → mtime → SHA-256 → 配置哈希），缓存键增加 `video_keyframe_interval` 和 `video_max_frames` 参数哈希。

### Q: 方案一和方案二能否共存？

可以。方案二检测到视频文件时走完整管道；方案一可以作为 `enable_video_deep_analysis=False` 时的轻量路径，此时 Parser 层做拆分，跳过 `VideoModalProcessor`。

## 七、BatchParser 批量处理中的音频支持

### 7.1 问题描述

`raganything/batch_parser.py` 中的 `BatchParser` 在批量解析时**没有注册音频格式**，导致音频文件在批量处理中被静默跳过。具体来说，存在三层缺失：

| 层级 | 位置 | 问题 |
|------|------|------|
| 扩展名过滤 | `batch_parser.py:101` — `get_supported_extensions()` | 返回的集合**未包含 `AUDIO_FORMATS`**，音频文件在 `filter_supported_files()` 阶段即被过滤 |
| 解析路由 | `batch_parser.py:158` — `process_single_file()` | 所有文件统一调用 `parser.parse_document()`，但各 parser 的 `parse_document()` 没有音频路由 |
| 后续处理 | `batch.py:306` — `process_documents_with_rag_batch()` | 两步流程中，第 1 步过滤掉音频，第 2 步的完整 RAG 处理（含 ASR）永远不会执行 |

### 7.2 各 Parser 的音频路由现状

| Parser | `parse_document()` 行为 | 处理音频结果 |
|--------|------------------------|-------------|
| `MineruParser` | 音频落到 `else` 分支 → 尝试作为 PDF 解析 | ❌ 失败 |
| `DoclingParser` | 音频落到 `else` 分支 → `raise ValueError` | ❌ 直接报错 |
| `PaddleOCRParser` | 音频落到 `else` 分支 → `raise ValueError` | ❌ 直接报错 |
| `Parser` 基类 | 提供 `parse_audio()` 方法，返回 `[{"type":"audio","audio_path":...,"file_name":...}]` | ✅ 可用 |

作为对比，`ProcessorMixin.parse_document()` (`processor.py:528-535`) 已正确按扩展名路由到 `doc_parser.parse_audio()`。

### 7.3 修复方案（最小修复）

采用**方案 A**：只改 `batch_parser.py`，不碰 parser 基类。

#### 改动 1: `get_supported_extensions()` 添加 AUDIO_FORMATS

在 `process_single_file()` 中添加按扩展名分发的逻辑，与 `ProcessorMixin.parse_document()` 保持一致。核心路由逻辑：

```python

    content_list = self.parser.parse_audio(
        audio_path=file_path,

        file_path=file_path,
```

#### 改动 3: 确保 batch.py 的批量 RAG 处理覆盖音频

`BatchMixin.process_documents_with_rag_batch()` 两步流程中，第 1 步返回成功解析的文件列表后，第 2 步调用 `process_document_complete()` 时，`ProcessorMixin` 已有完整的音频处理逻辑（含 ASR 转录和多模态处理），无需额外改动。

### 7.4 数据流对比

修复前后音频文件的处理路径对比：

**修复前：**
```
音频.mp3 → filter_supported_files() → 不支持扩展名 → 跳过 ❌
```

**修复后（仅 BatchParser）：**
```
音频.mp3 → filter_supported_files() → 通过 ✅
          → process_single_file()
              → parse_audio() → [{"type":"audio","audio_path":"...","file_name":"..."}]
              → 记录成功解析
```

**修复后 + process_documents_with_rag_batch() 完整链路：**
```
音频.mp3 → BatchParser 解析成功 ✅
          → process_document_complete() (ProcessorMixin)
              → parse_document() 路由到 parse_audio()
              → AsrModalProcessor ASR 转录
              → 实体提取 → 知识图谱插入 ✅
```

### 7.5 注意事项

1. **BatchParser 的角色边界**：`BatchParser` 只做纯解析（记录音频路径），ASR 转录依赖后续的 `ProcessorMixin`。这是设计使然，不是缺陷。
2. **缓存一致性**：`BatchParser` 不做缓存管理（由 `ProcessorMixin` 的 `_store_cached_result()` 负责），两阶段流程下缓存仍然有效。
3. **CLI 支持**：`batch_parser.py` 的 CLI 入口 `main()` 无需额外修改，只要 parser 支持音频解析，CLI 自动受益。

#### 添加 `parse_xxx()` 基类方法

Parser 层保持轻量——**只做路径记录，不做实际处理**。实际处理由 ModalProcessor 完成。

```python
def parse_video(
    self,
    video_path: Union[str, Path],
    output_dir: Optional[str] = None,
    **kwargs,
) -> List[Dict[str, Any]]:
    """
    Parse video file — minimal: just records the path.
    All processing is done by VideoModalProcessor.

    Supported formats: {VIDEO_FORMATS}
    """
    video_path = Path(video_path)
    if not video_path.exists():
        raise FileNotFoundError(f"Video file does not exist: {video_path}")

    ext = video_path.suffix.lower()
    if ext not in self.VIDEO_FORMATS:
        raise ValueError(
            f"Unsupported video format: {ext}. "
            f"Supported formats: {', '.join(sorted(self.VIDEO_FORMATS))}"
        )

    return [{
        "type": "video",
        "video_path": str(video_path.resolve()),
        "file_name": video_path.name,
    }]
```

**返回格式** — 所有 `parse_xxx()` 返回统一的 `List[Dict]`：

```python
# 通用格式
[
    {
        "type": "<类型标识符>",     # 必须，与 _PROCESSOR_KEY_MAP 的 key 一致
        "<path_field>": "<绝对路径>", # 主内容路径
        "file_name": "<文件名>",      # 原始文件名
        # 可选附加元数据...
    }
]
```

各类型的具体字段约定：

| type | path_field | 说明 |
|------|-----------|------|
| `"audio"` | `audio_path` | 音频文件绝对路径 |
| `"video"` | `video_path` | 视频文件绝对路径 |
| `"image"` | `img_path` | 图片绝对路径 |
| `"text"` | `text` | 纯文本内容（不走 multimodal 管道） |
| `"table"` | `table_body` | 表格数据（markdown 或 dict） |
| `"equation"` | `latex` / `text` | LaTeX 公式或文本描述 |
| 自定义 | 自定义字段 | 扩展类型可自由定义 |

```python
    # 以下为 processor.py 内部用到，可选但不推荐省略：
    "chunk_id": "chunk-xxxx",                  # 生成的 chunk ID
    "content_type": "video",                   # 与 content_type 一致
}
```

#### process_multimodal_content 返回值详解

```python
# Tuple[enhanced_caption: str, entity_info: Dict, chunk_results: List]

# enhanced_caption — 最终增强描述文本（与 description 类似或相同）
# entity_info — 同 generate_description_only
# chunk_results — LightRAG 格式的 chunk 结果列表，每项为 (maybe_nodes, maybe_edges) 元组
chunk_results = [
    (
        {"entity_name": {"node_data": ...}},    # maybe_nodes: Dict[str, List[Dict]]
        {("src", "tgt"): [{"relation": ...}]},   # maybe_edges: Dict[Tuple, List[Dict]]
    )
]
```

### 8.8 Step 7: 初始化 — `raganything.py`

在 `RAGAnything.__post_init__()` 或等效的初始化入口中注册 processor：

```python

if self.config.enable_video_processing:
    from raganything.video_processor import VideoModalProcessor  # 或从 modalprocessors 导入

    self.modal_processors["video"] = VideoModalProcessor(
```

### 8.9 接入 Checklist

| # | 文件 | 改动 | 必须? |
|---|------|------|--------|
| 1 | `parser.py` | 添加格式常量集（如 `VIDEO_FORMATS`） | ✅ 必须 |
| 1b | `parser.py` | 添加 `parse_xxx()` 基类方法 | ✅ 必须 |
| 2 | `processor.py` | `parse_document()` 中添加扩展名路由 | ✅ 必须 |
| 3 | `utils.py` | `_PROCESSOR_KEY_MAP` 注册新 type | ✅ 必须 |
| 4 | `config.py` | enable 开关 + 参数（可选） | ❌ 可选，建议 |
| 5 | `modalprocessors.py` 或新文件 | 新建 Processor 类 | ✅ 必须（至少 `generate_description_only`） |
| 6 | `prompt.py` | chunk 模板 | ✅ 必须 |
| 6b | `processor.py` | `_apply_chunk_template()` 新类型分支 | ✅ 必须 |
| 7 | `raganything.py` | 注册 processor 到 `modal_processors` | ✅ 必须 |
| — | `batch_parser.py` | **无**需改动，自动覆盖 | — |

**特别说明**：

- **1b 与 2 的关系**：`parse_xxx()` 基类方法定义解析逻辑，`processor.py` 的路由决定何时调用它。两者缺一不可。
- **最小可行接入**：如果只做 Parser 层的解析（不做知识图谱插入），只需要 1 + 2 + `batch_parser.py`（自动）。但完整的 RAG 链路需要全部 7 步。
- **GenericModalProcessor fallback**：如果 `_PROCESSOR_KEY_MAP` 注册了新类型但 `modal_processors` 中没有对应的 processor，系统会 fallback 到 `GenericModalProcessor`，它只做基本的描述生成，不会出错但精度有限。

### 8.10 实例：Whisper 音频接入全链路

以已实现的 Whisper 音频接入为例，串联全部步骤：

```
meeting.mp3
  │
  ├── [Step 1 — 格式声明]         parser.py Parser 基类
  │                               AUDIO_FORMATS = {".mp3", ".wav", ".m4a", ...}
  │                               parse_audio() → [{"type":"audio","audio_path":"...","file_name":"..."}]
  │
  ├── [Step 2 — Parser 路由]      processor.py parse_document()
  │                               ext in Parser.AUDIO_FORMATS → doc_parser.parse_audio()
  │                               batch_parser.py: 自动路由到 parse_audio()
  │
  ├── [Step 3 — Content 路由]     utils.py
  │                               _PROCESSOR_KEY_MAP["audio"] = "audio"
  │                               separate_content() → type="audio" 归入 multimodal_items
  │
  ├── [Step 4 — Config]           config.py
  │                               enable_audio_processing: bool = True
  │                               asr_binding: str = "whisper"
  │                               asr_model: str = "large-v3"
  │
  ├── [Step 5 — ModalProcessor]   modalprocessors.py — AsrModalProcessor
  │   ┌────────────────────────────────────────────────────────────────┐
  │   │ generate_description_only()                                    │
  │   │   ├─ _load_or_transcribe()  → ASR 转录（本地 Whisper 模型）     │
  │   │   ├─ _segment_with_overlap() → 按 token 分段（含 overlap）      │
  │   │   └─ _recursive_mapreduce()  → 逐层合并摘要 → 实体提取          │
  │   │                                                               │
  │   │ 返回: (description: str, entity_info: Dict)                   │
  │   │   description = "会议讨论了Q2营收..."                          │
  │   │   entity_info = {                                             │
  │   │     "entity_name": "Q2营收会议 (audio)",                      │
  │   │     "entity_type": "audio",                                   │
  │   │     "summary": "...",                                         │
  │   │   }                                                           │
  │   └────────────────────────────────────────────────────────────────┘
  │
  ├── [Step 6 — Chunk 模板]       prompt.py PROMPTS["audio_chunk"]
  │                               processor.py _apply_chunk_template()
  │                               按 content_type == "audio" 路由 → 音频模板
  │                               每个 segment 生成一个独立 chunk（含时间戳、说话人标签）
  │
  ├── [Step 7 — 初始化注册]        raganything.py
  │                               self.modal_processors["audio"] = AsrModalProcessor(...)
  │
  ├── [ASR 后端抽象层]             raganything/asr/ 目录
  │                               AsrBackend 基类 + WhisperBackend（默认）
  │                               @register_asr_backend() → "whisper"
  │                               可选切换: ASR_BINDING=openai_compatible → OpenAI 兼容 API
  │
  └── [入库]                      processor.py _process_multimodal_content()
                                  每个 segment → 独立 chunk → 实体提取 → belongs_to 关系 → 知识图谱
```

**关键实现细节**（audio 特有的处理）：

- **ASR 后端抽象**：`AsrBackend` 基类定义了 `transcribe()` 接口，`WhisperBackend` 和 `OpenAICompatibleBackend` 是两个内置实现。切换后端只需改 `.env` 中的 `ASR_BINDING`，无需改代码。
- **多 chunk 生成**：一个音频文件 → N 个 transcript segment → N 个 chunk（带 overlap），每个 chunk 含：时间戳、转录文本、说话人标签、组摘要。
- **缓存策略**：3 层验证：文件大小 → mtime → SHA-256 + 配置哈希，按 `asr_identity` 缓存。
- **MapReduce 摘要**：segment 列表 → recursive reduce → 全局摘要 → entity 提取。
